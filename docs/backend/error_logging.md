# 에러 로깅 및 모니터링 전략 (Error Logging & Monitoring)

## 1. 로깅 아키텍처 개요

미쿠의 모든 에러와 이벤트를 체계적으로 기록하여, 문제 발생 시 원인을 빠르게 파악하고 시스템 안정성을 유지합니다.

## 2. 로그 레벨 및 분류

### 2.1. 로그 레벨 (Log Levels)
-   **DEBUG**: 개발 중 상세 정보 (함수 진입/종료, 변수 값).
-   **INFO**: 정상 동작 이벤트 (대화 시작, 모델 로딩 완료).
-   **WARNING**: 잠재적 문제 (응답 지연, VRAM 사용량 높음).
-   **ERROR**: 에러 발생 (LLM 추론 실패, GPU 오류).
-   **CRITICAL**: 시스템 중단 위험 (메모리 부족, 하드웨어 오류).

### 2.2. 로그 카테고리 (Categories)
-   **LLM**: Gemma 추론, 컨텍스트 로딩, 토큰 생성 관련.
-   **TTS/STT**: 음성 처리 파이프라인 오류.
-   **Vision**: 웹캠/화면 캡처, 객체 인식 실패.
-   **GPU**: CUDA 오류, VRAM 부족, GPU 온도 경고.
-   **Network**: WebSocket 연결 끊김, API 타임아웃.
-   **DB**: 데이터베이스 쿼리 실패, 벡터 검색 오류.
-   **Plugin**: 플러그인 실행 오류, 권한 위반.
-   **Frontend**: 렌더링 오류, 메모리 누수 의심.

## 3. 로그 저장 전략

### 3.1. 저장 위치 및 보관 기간

| 로그 레벨 | 저장 위치 | 보관 기간 | 용도 |
| :--- | :--- | :--- | :--- |
| **DEBUG** | 메모리 (RAM) | 실시간만 (파일 미저장) | 개발 중 디버깅 |
| **INFO** | SSD (C:/MIKU_DATA/logs/info/) | 7일 | 일반 동작 추적 |
| **WARNING** | SSD (C:/MIKU_DATA/logs/warning/) | 30일 | 패턴 분석 |
| **ERROR** | SSD (C:/MIKU_DATA/logs/error/) | 90일 | 문제 해결 |
| **CRITICAL** | HDD (D:/MIKU_DATA/logs/critical/) | 영구 보관 | 시스템 이슈 기록 |

### 3.2. 로그 파일 형식
-   **파일명**: `{category}_{date}_{hour}.log` (예: `llm_2026-02-08_14.log`)
-   **로테이션**: 시간별 파일 분할 (용량 관리).
-   **포맷**: JSON Lines (JSONL) - 구조화된 로그로 파싱 용이.
-   **선택 이유**: JSON 형식으로 저장하여 파싱 및 분석이 편리함.

```json
{
  "timestamp": "2026-02-08T14:30:25.123Z",
  "level": "ERROR",
  "category": "LLM",
  "service": "gemma_inference",
  "message": "CUDA out of memory",
  "context": {
    "model": "gemma-4-12b-4bit",
    "context_length": 4096,
    "vram_used": "15.8GB",
    "vram_total": "16GB"
  },
  "stack_trace": "...",
  "user_id": "default"
}
```

## 4. 에러 처리 전략

### 4.1. LLM 추론 실패
-   **재시도**: 최대 3회 (지수 백오프: 1초, 2초, 4초).
-   **Fallback**: 실패 시 Emergency Model (Qwen 1.5B)로 전환.
-   **로그**: ERROR 레벨로 기록 + 사용자에게 "잠깐만, 머리가 띵해..." 알림.

### 4.2. GPU 오류 (CUDA Error)
-   **자동 복구**: GPU 프로세스 재시작, 모델 재로딩.
-   **로그**: CRITICAL 레벨로 기록 + 시스템 알림.
-   **사용자 피드백**: "오빠, 나 머리가 너무 뜨거워..." (온도 경고와 연동).

### 4.3. WebSocket 연결 끊김
-   **재연결**: 자동 재연결 (최대 10회, 5초 간격).
-   **로그**: WARNING 레벨 (정상적인 네트워크 이슈).
-   **상태 복구**: 재연결 시 대화 컨텍스트 복원.

### 4.4. 플러그인 오류
-   **격리**: 플러그인 프로세스 크래시가 메인 시스템에 영향 없도록.
-   **로그**: ERROR 레벨 + 플러그인 이름 기록.
-   **복구**: 플러그인 비활성화 후 메인 시스템은 정상 동작.

## 5. 로그 모니터링 및 알림

### 5.1. 실시간 모니터링
-   **로그 집계**: 최근 1시간 내 ERROR/CRITICAL 발생 횟수 추적.
-   **임계값**: 10분 내 ERROR 5회 이상 -> 미쿠가 직접 알림 ("뭔가 이상해...").

### 5.2. 로그 분석 도구
-   **Python**: `structlog` (구조화된 로깅), `loguru` (간편한 설정).
-   **시각화**: (선택) Grafana 대시보드 (에러 트렌드, 카테고리별 분포).

## 6. 개인화 및 프라이버시

-   **로컬 전용**: 모든 로그는 로컬에만 저장 (외부 전송 없음).
-   **민감 정보 필터링**: (선택) 비밀번호, API 키 등은 자동 마스킹.
-   **사용자 제어**: 로그 삭제 권한 (D드라이브 백업 제외).

## 7. 디버그 모드

-   **활성화**: 환경 변수 `MIKU_DEBUG=true` 또는 설정 파일.
-   **동작**: 화면 구석에 Chain of Thought (사고 과정) 텍스트 출력.
-   **로그**: DEBUG 레벨도 파일에 저장 (일반 모드에서는 메모리만).
